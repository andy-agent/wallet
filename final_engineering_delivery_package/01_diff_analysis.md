# 01_diff_analysis

## 1. 三套工程包总体定位差异
| 包 | 文件 | 定位 | 优点 | 短板 |
|---|---|---|---|---|
| Package A | CryptoVPN_Consolidated_v2_Package.zip | 业务结论收敛包 | 对钱包、多链支付、单活跃 session、25%/5% 分佣、USDT on Solana 提现等业务结论收敛明确 | 工程资产不够全；页面/Admin/测试/部署粒度不足；FastAPI 参考路径已落后于本轮技术基线 |
| Package B | CryptoVPN_Engineering_Audit_Rebuild_v3_Package.zip | 审计驱动的工程重构包 | 审计与冲突识别完整；OpenAPI 最完整；DDL 与状态机较成熟；适合做主收敛基底 | 后端技术栈在 FastAPI / NestJS 之间未定；钱包广播边界偏服务端化；部分支付策略仍保留 TBD |
| Package C | cryptovpn_engineering_delivery_v3.zip | 工程交付执行包 | Android/Admin 页面、任务拆解、工程骨架、测试与 Kimi 任务包更完整，最贴近开发排产 | 冲突裁决不如 Package B 完整；部分命名漂移；FastAPI/Alembic 偏向与本轮最终基线冲突 |
| 辅助业务基线 | crypto_vpn_total_prd_v1_1.md | 已确认业务冻结规则 | 邮箱优先、单活跃终端、SOL/USDT(Solana)/USDT(TRC20)、一级25%/二级5%、最低10 USDT、默认提现 USDT on Solana | 不是完整工程包，不能单独充当最终开发启动包 |

## 2. 各包优点
### Package A
- 业务冻结结论最清晰，尤其是单活跃 session、支付范围、25%/5% 分佣、默认提现规则。
- 适合作为 **业务基线校准文件**。

### Package B
- 审计意识最强，冲突识别更系统。
- OpenAPI、DDL、状态机、任务链路覆盖更完整。
- 适合作为 **工程契约主骨架**。

### Package C
- Android/Admin 页面、测试、工程骨架、任务拆解、Kimi 任务包更适合实际排产。
- 适合作为 **执行层补强资产**。

## 3. 各包缺点
### Package A
- 工程层粒度不足；Admin、测试、部署与 CI/CD 不够完整。
- 仍带有历史 FastAPI 倾向，不适合作为唯一最终版。

### Package B
- 技术栈裁决不彻底；FastAPI / NestJS 留有并行口径。
- 钱包广播边界偏服务端，和最新冻结规则存在偏差。

### Package C
- 冲突裁决与来源追溯不如 Package B 完整。
- 部分命名与 DDL/API 存在轻微漂移，如 account_devices / account_installations、wallet_transactions 是否保留等。

## 4. 共识项
1. 产品核心是 VPN 订阅，而不是钱包或交易所。
2. 钱包必须是自托管，不允许服务端保存私钥与助记词。
3. 首期支付支持链上支付，且支付确认后自动开通订阅。
4. 单账号需控制共享风险。
5. 官方节点 + 区域选择 + 不允许导入外部节点。
6. 分佣采用平台账本而不是实时链上拆分。
7. 后台必须覆盖用户、订单、套餐、节点、提现、版本、审计。
8. 整个系统必须用状态机和幂等策略驱动。

## 5. 重复项
- 账号/订阅/订单/支付/分佣/提现/版本/法务等主领域对象，在 3 包中均已出现。
- Android 端 Tab 结构、订单主链路、邀请提现闭环在 3 包中都已被定义。
- OpenAPI 与 DDL 在 B/C 中均已具备较高成熟度，可反向校正 PRD。

## 6. 主要冲突项
1. 后端技术栈：FastAPI vs NestJS vs Go。
2. 钱包边界：轻支付壳 vs 自托管轻钱包 vs 完整多链钱包的 MVP 表达方式。
3. 设备控制：显式设备绑定 vs 单活跃 session。
4. 支付资产范围：是否包含 USDT on Solana。
5. 支付目标策略：专属地址 vs 共享地址 + 唯一金额。
6. 钱包广播：客户端直连 vs 服务端代理主导。
7. 计划-区域建模：JSON 快照 vs 关系映射。
8. 订阅的 expiring_soon 是否持久化。
9. 服务端是否保存钱包元数据与交易历史。
10. 部署基线：K8s vs Docker Compose。

## 7. 缺失项
1. 缺少统一主入口文件，历史包均未形成唯一消费顺序。
2. 缺少唯一版“最终事实源”说明。
3. 缺少多 Agent 编排层面的开发入口。
4. 缺少“PRD/API/DDL/页面/测试/部署”一体化一致性总报告。
5. 对“客户端直连广播、后端兜底广播”的最新冻结边界没有完整落到所有历史资产中。

## 8. 风险项
- 历史包若直接拼接，会出现命名漂移和状态不闭环。
- 继续保留 FastAPI 与 NestJS 并行口径，会导致 Agent 任务拆分失效。
- 钱包广播边界不统一，会导致安全模型和 Android/Backend 联调目标冲突。
- 支付目标策略不统一，会让 DDL、Scanner、收银台和测试全部分叉。
- 若 expiring_soon 落库，状态机会引入不必要的定时写扩散。

## 9. 推荐采用策略
### 9.1 收敛原则
- 业务冻结结论：以 v1.1 与用户最新明确补充为准。
- 工程契约：以 Package B 为主骨架。
- 页面、任务、执行与落地性：以 Package C 为补强。
- 已过时或与最新规则冲突的内容：删除而不是保留多版本并列。

### 9.2 采用方式
- **A 包**：吸收业务冻结事实，不直接作为工程主包。
- **B 包**：吸收审计、OpenAPI、DDL、状态机思路。
- **C 包**：吸收页面规格、任务拆解、工程骨架、Kimi 任务包。

## 10. 收敛策略说明
本次最终包采用以下方法：
1. 先做差异分析与冲突裁决。
2. 再统一术语、目录、工程边界。
3. 再重写唯一版 PRD、API、DDL、页面、状态机、测试、任务、部署。
4. 最后出一份一致性自检报告，确保不再出现“PRD 有、API 无、DDL 无、页面无入口”的断裂。

## 11. 最终推荐结论
- **业务基线**：采用 v1.1 冻结规则。
- **工程收敛基底**：采用 Package B。
- **执行与多 Agent 视角补强**：采用 Package C。
- **最终产物**：本目录下的 `00_master_delivery_index.md` 到 `16_consistency_check_report.md`。
