## 当前建议分工

### 方案 A

- 服务器一 `38.58.59.119`
  - 主后端节点
  - PostgreSQL
  - Redis
  - NestJS backend
  - 后续 worker
- 服务器二 `38.58.59.142`
  - Admin Web
  - 官网下载页
  - Nginx / 证书接入
- 服务器三 `154.37.208.72`
  - VPN control / 节点控制相关组件
  - 预留链上辅助任务、监控或灰度实验用途

### 当前采用理由

- 先把有状态组件集中在一台机器，减少早期联调复杂度
- 把面向外部访问的 Web 面和 API 面拆开，降低反向代理调整成本
- 给 VPN / 节点控制与后续链上辅助逻辑预留独立位置，避免过早和主后端耦合

### 后续可调整点

- 若服务器一压力过高，可将 PostgreSQL / Redis 单独拆到服务器三
- 若只做最小真实环境测试，可先在服务器一单机跑通 backend + postgres + redis，再逐步拆分
