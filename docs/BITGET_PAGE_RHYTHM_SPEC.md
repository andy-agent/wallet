# Bitget 页面节奏与布局规则

本文只基于 master 上两份源文档归纳页面结构与节奏规则：

- `/Users/cnyirui/git/projects/liaojiang/docs/BITGET_CAPTURE_INVENTORY.md`
- `/Users/cnyirui/git/projects/liaojiang/docs/BITGET_PAGE_MATRIX.md`

颜色、圆角、底栏、sheet 等共享 token 只做 brief 引用，具体值仍以页面矩阵中的统一基线为准；本文只解决 4j0.22.3 / 4j0.22.4 / 4j0.22.5 需要复用的页面族节奏。

## 1. 首页族节奏（供 4j0.22.3）

### 1.1 首页默认态的主栈顺序

- 首页必须保持单列纵向阅读，首屏顺序固定为：顶部搜索 chrome -> 资产总览 KPI 区 -> 4 个快捷入口 -> 运营主卡 -> 次级卡片流 -> 悬浮底部导航。
- 顶部搜索 chrome 只承担进入与分发，不承担解释性文案；主信息密度从资产总览区开始。
- 资产总览区是首页唯一的首屏 hero，必须先出现主数值，再出现盈亏或状态，再出现单个高优先级 CTA。
- 首页首屏 KPI 区严格遵守“主数值一行 -> 变化值/状态一行 -> 单主 CTA 一行”的三段节奏，不在这一块并排塞入第二个主按钮。
- 快捷入口只占一行，优先做成等权重胶囊或图标入口，不在这一层混入副文案。
- 首页卡片流采用“一张主卡带图 + 连续次级卡片”的节奏，不做双列瀑布流，也不在首屏连续堆叠多个同等级大 Hero。

### 1.2 信息流与卡片堆叠规则

- 首张运营卡必须是视觉最重的整宽卡，卡内自带标题、说明和按钮，承担首个内容转化位。
- 第二张开始切换为纵向单列卡片流，继续保留大圆角，但降低视觉重量，避免每张都像 Hero。
- 若借用“理财 landing”语法，顺序应为：顶端主题 Hero -> Hero 内嵌信息条 -> 横向卡组 -> 纵向列表卡；不要把横滑卡组放到页面底部。
- 首页列表节奏应优先支持“1 个大卡 + 1 至 3 个普通卡可见”的首屏密度，保证用户向下滑动前就能理解页面层次。

### 1.3 首页 overlay / modal

- “全部功能”是从首页主栈上方升起的动作总表，不是独立新页；背景首页必须继续可辨认。
- 动作总表的节奏固定为：拖拽条 -> 分组标题 -> 宫格组 -> 下一分组标题 -> 下一组入口，不在底部再叠加额外说明区。
- 首页确认类弹层采用“背景业务页 + 前景底部确认 sheet”模式；背景必须保留当前业务上下文，不能切成纯黑背景。
- 确认 sheet 固定为标题、正文、双按钮三段，不在正文中再混入表单或列表。

### 1.4 给 4j0.22.3 的直接落地要求

- `app_shell?tab=home` 先实现“资产总览 + 4 快捷入口 + 单列卡片流”的阅读节奏，再替换具体业务文案。
- 套餐推荐或活动专区优先借用“理财 landing”的 Hero + 横滑卡 + 列表卡顺序，不要为每个专题重新发明版式。
- 首页动作总表与钱包支付确认都应复用底部 sheet 语法，而不是另做全屏弹窗。

## 2. 钱包总览与交易节奏（供 4j0.22.3）

### 2.1 钱包总览页的固定堆叠

- 钱包首页主栈顺序固定为：顶部搜索 chrome -> 当前钱包身份行 -> 资产总额与盈亏 -> 3 个快捷动作 -> 2 张业务卡 -> 资产列表标题行 -> 资产列表。
- 身份行必须位于资产总额之前，因为该页先表达“当前看的是哪个钱包”，再表达“这个钱包有什么资产”。
- 快捷动作必须保持同层同权重的 3 按钮横排，不拆到列表项内，也不塞进顶部菜单。
- 中段业务卡保持双卡并列节奏，用来承接支付、理财、权益等次级入口；不要把这两张卡拆成纵向长列表。

### 2.2 列表节奏与空状态

- 资产列表标题行必须同时承载分区名称和筛选/排序入口，说明列表是本页主体，不是附属模块。
- 资产列表每行采用高信息密度双侧布局：左侧资产身份与涨跌，右侧持仓数量与法币折算。
- 空仓态不切换为独立插画空页面；必须沿用同一 chrome、同一资产总览区、同一快捷动作和同一双卡结构，只把列表内容降为零资产行。
- “代币管理”类入口应出现在资产列表尾部，作为列表的收尾动作，而不是在首屏抢占主层级。

### 2.3 钱包切换器 overlay

- 钱包切换器是选择器，不是设置页；节奏固定为：标题栏 -> 总资产摘要 -> 当前钱包高亮卡 -> 其他钱包卡 -> 底部单主 CTA。
- 当前钱包卡必须在视觉上先于其他钱包项出现，并通过高亮边框而非额外说明文字来表达选中态。
- 底部 `添加钱包` 是该 sheet 唯一强操作；卡片点击本身负责切换，不要再给每张卡单独放次级按钮。

### 2.4 资产详情、收款、转账

- 资产详情页的头部结构固定为：资产图标与名称/链名 -> 主数量与法币等值 -> 头部主 CTA；随后才进入行情卡、历史列表和底部动作。
- 这个 detail header 只服务单个资产对象，头部同层只保留一个高优先级 CTA；`收款/转账` 这类闭环动作统一下沉到底部动作区。
- 详情页中的 `链上行情` 与右侧型 KPI 规则应采用“标签在左、值在右”的信息表语法，避免变成说明文段落。
- 交易历史必须作为详情页主内容区之一，位置应在头部摘要之后，不能被收起到二级页。
- 详情页底部动作保持 `收款/转账` 成对出现，形成该页的主闭环；`安全检测` 属于弱层级尾部入口。
- 收款页采用“接近全屏弹出页”的节奏：关闭按钮 -> 中央资产图标 -> 标题与网络说明 -> 大二维码 -> 地址卡 -> 提示条 -> 底部单主 CTA。
- 收款页的二维码必须是主视觉中心，地址卡和提示条都只能服务二维码，不应反客为主。
- 转账地址输入页采用“输入优先”结构：标题 -> 大输入框 -> 两个快捷按钮 -> 页内 tabs -> 空状态或候选内容。
- 转账页初始态的 CTA 不应抢在输入区前面出现；在无有效地址时，页面重心应停留在输入与来源选择上。

### 2.5 给 4j0.22.3 的直接落地要求

- `wallet_home` 先复刻固定堆叠，再分别补有持仓态与空仓态；两者只能换内容，不能换布局。
- `asset_detail/{assetId}` 需要先做头部摘要、KPI/信息卡、历史列表、底部双动作的完整顺序，再填具体业务数据。
- `receive` 与 `send` 必须共享同一族的标题区、按钮节奏和结果反馈语言，避免变成两套风格。

## 3. VPN 替代页族节奏（供 4j0.22.4）

### 3.1 详情型页面：由“行情详情页”转译

- VPN 详情型页面必须沿用“返回 + 标题/副标题 + tabs + 主 KPI + 侧边 KPI 列 + 主可视化区 + 底部主 CTA”的顺序。
- 顶部标题区要同时容纳主标题与副标题，说明这是“对象详情”而不是纯列表页。
- `行情/详情` 的双 tabs 说明详情页内部仍有语义分区；迁移到 VPN 后也应保留顶部二段切换，而不是塞回一个长页面里。
- 主 KPI 区必须先给出一个最大的核心数值或状态，再用右侧一列承载 4 到 5 个关键指标；这组指标应右对齐并保持表格式节奏。
- 详情型 VPN 页的 CTA 只能有一个最终落点：头部若已有对象级动作，则页面底部仍需保留唯一的主转化按钮，不能在中段再插入第三层主 CTA。
- K 线图所在的大可视化区，在 VPN 里可以替换为套餐表现图、节点质量图或订单状态图，但必须保留“页面中段主视觉 + 下方指标切换或附属说明”的版式角色。
- 底部主 CTA 始终是该详情页的最终落点，不应上移到头部 KPI 旁边。

### 3.2 选择/下单型页面：由“跨链闪兑”转译

- VPN 选择与下单页应采用“标题栏 -> 风险/说明提示条 -> 上下双卡 -> 中间切换控件 -> 单主 CTA -> 底部模式 dock”的顺序。
- 上下双卡是这类页面的骨架：上卡表达当前选择或输入，下卡表达目标选择或结果，不应被改写成普通表单长列表。
- 中间圆形切换按钮必须位于双卡之间，表达可交换或可反转的关系；即使业务不完全对称，也要保留这个节奏点。
- 主 CTA 要紧跟双卡之后，形成“选完即提交”的操作闭环；不要把 CTA 下沉到很长的说明区之后。
- 底部工具 dock 属于模式切换层，不是系统底部导航；在 VPN 中可以映射为套餐、区域、订单、支付等子模式，但应继续保持与主内容分层。

### 3.3 选择器页：由“选择网络”转译

- 区域/节点选择器采用全屏选择页而不是底部 sheet，顺序固定为：关闭 -> 标题 -> 搜索框 -> 单选列表 -> 底部单主 CTA。
- 单选列表必须整行可点，选中态只通过右侧对勾和当前高亮表达，不再额外出现二级按钮。
- 搜索框必须放在列表之前，说明过滤是一级动作；不要把筛选藏进右上角弹出菜单。

### 3.4 信息表与详情头部的共用规则

- 从“行情详情页”与“理财产品详情”共同提炼出的规则是：详情头部负责核心摘要，信息表负责稳定字段，不要把固定字段写成多段说明文。
- 右侧 KPI 列与 `产品信息` 区都应采用 key/value 结构；左侧标签短、右侧值右对齐，必要时允许复制按钮或安全提示胶囊附着在值附近。
- 单条曲线或单个主图只能服务一个对象详情；多个并列图表会破坏当前 family 的节奏。

### 3.5 给 4j0.22.4 的直接落地要求

- `region_selection` 应先落地，因为它定义了搜索、单选列表和底部主按钮的统一选择器语法。
- `vpn_home` / `plans` / `order_checkout/{planId}` 应统一复用“双卡 + 中间切换 + CTA + 模式 dock”的节奏，不为每个步骤单独设计表单结构。
- `order_detail/{orderId}` 以及套餐详情必须复用“主 KPI + 侧边 KPI 列 + 信息表 + 底部 CTA”的详情型结构。

## 4. Discover / Growth / Legal 容器节奏（供 4j0.22.5）

### 4.1 容器与加载首帧

- Discover/Growth/Legal 这组页面的最小容器只有返回箭头和深色背景，说明首帧骨架要极简，不能预先放置错误的栏目标题或假列表。
- 对 WebView 或远端内容承载页，应先展示极简容器，再进入正式内容；不要在加载期复用 Profile 列表样式。
- 空白首帧表达的是“容器尚未装载完成”，不是“没有内容”；因此不应放中心插画空状态。

### 4.2 增长页的 Hero 与内容节奏

- 已加载的邀请中心采用“顶部返回/辅助动作 -> 大标题与副标题 -> 首屏 Hero 插画 -> 表单卡 -> tabs 卡 -> 规则与分享内容”的顺序。
- Hero 必须位于表单卡之前，用来先解释页面主题，再承接邀请码或奖励创建行为。
- 表单卡是首个操作密集区，应保持大圆角独立卡片身份，不要拆散到 Hero 或长表单里。
- `推荐奖励` 这类 tabs 卡位于表单卡之后，说明“先建立用户入口，再展示规则分发”，不要把 tabs 提前到首屏最顶。
- WebView 承载的长内容区应作为下半页连续滚动内容，和前面的 Hero/表单卡保持同一纵向流，不要做嵌套滚动区域。

### 4.3 Legal / Commission / Withdraw 的派生规则

- 缺少独立截图的法务、佣金、提现页，应先复用“极简容器 + 单个首屏摘要区 + 后续连续内容区”的节奏，不要套用首页 feed 或 Profile 设置列表。
- 法务文档详情优先采用“简单顶部 chrome + 长文内容容器”结构，CTA 应该少而后置。
- 佣金台账与提现页若需要摘要信息，摘要卡应先于明细列表出现，但不能强过增长页 Hero。

### 4.4 给 4j0.22.5 的直接落地要求

- `app_shell?tab=discover` 先实现容器首帧与已加载增长页两种状态，不要跳过加载壳层。
- `invite_center` 必须优先实现 Hero、邀请码表单卡、奖励 tabs 卡、长内容区四段顺序。
- `commission_ledger`、`withdraw`、`legal_documents`、`legal_document_detail/{documentId}` 在缺图状态下只允许沿用同 family 的容器节奏派生，不新增全新视觉层级。

## 5. Profile / Settings 层级节奏（供 4j0.22.5）

### 5.1 设置主页层级

- Profile 主页不是个人主页 feed，而是设置总表；顺序固定为：顶部标题栏 -> 单列设置卡列表 -> 关于卡 -> 底部帮助入口。
- 设置卡列表必须是主内容区，条目按“安全/地址/消息/偏好/线路/增长/绑定”这类功能层级顺序向下展开。
- `关于 Bitget Wallet` 作为独立卡片晚于主列表出现，说明版本和品牌信息属于次级层。
- 底部两个帮助入口是页面尾声动作，位置应独立于主设置卡列表。

### 5.2 钱包安全页

- 钱包安全页采用“身份摘要 -> 风险等级提示 -> 高亮安全设置卡 -> 普通列表项 -> 危险操作按钮”的顺序。
- 身份摘要区必须居中，并与设置列表语法断开，说明这里先展示账户本体，再展示配置项。
- 风险等级与 `安全设置` 高亮卡应连续出现，形成安全页的核心提醒区；不要把提醒拆散到多个列表项里。
- `退出钱包` 作为危险操作必须单独沉到底部，与普通列表项留出明显节奏断层。

### 5.3 头像选择器 overlay

- 头像选择器是近全屏的高层 overlay，顺序固定为：拖拽条 -> 大头像预览 -> 主题色圆点 -> 头像分组宫格 -> 底部保存按钮。
- 大头像预览必须先于色板和宫格出现，否则用户无法理解当前正在编辑的对象。
- 主题色圆点是对预览的即时修饰层，不能下沉到宫格分组之后。
- `推荐` 与 `笑脸与人物` 等分组应使用连续纵向宫格，而不是分页切换。
- `保存` 必须固定在底部，且只有内容真正变化后才进入可点击态。

### 5.4 给 4j0.22.5 的直接落地要求

- `profile` 主页只做设置总表，不做社交主页式头图或资产概览。
- Profile 安全子页要先完成身份摘要和高亮安全卡，再补足列表项与危险按钮。
- 头像或主题选择器必须走高层 sheet，不要做成独立新页。

## 6. Overlay / Modal 分层总规则

| 类型 | 来源页面族 | 典型结构 | CTA 规则 |
|---|---|---|---|
| 动作总表 sheet | Home 全部功能 | 拖拽条 -> 分组标题 -> 宫格组 -> 下一分组 | 可无 footer CTA，动作由宫格项直接触发 |
| 选择器 sheet | Wallet 选择钱包 | 标题栏 -> 摘要 -> 选项卡片 -> 底部主 CTA | 只保留 1 个底部主 CTA |
| 确认 sheet | Home 授权确认 | 标题 -> 正文 -> 双按钮 | 仅确认类使用左右双按钮 |
| 高层 picker sheet | Profile 头像选择器 | 预览 -> 色板 -> 宫格 -> 底部保存 | 底部保存固定，按变更状态启用 |
| 全屏选择页 | VPN 选择网络 | 顶部关闭/标题 -> 搜索 -> 单选列表 -> 底部主 CTA | 列表点击负责选择，底部 CTA 负责提交 |

- 所有 overlay 都必须保留父页面上下文可辨性，至少让用户知道“自己是从哪一层进来的”。
- 弹层推进顺序优先保持为“父页 -> 选择器/动作总表 -> 确认 modal -> 结果页或返回父页”，不要在单次任务流里连续叠三层以上 sheet。
- sheet 的滚动内容和底部 CTA 必须分层；CTA 不应跟随内容滚出屏幕。
- 弱层级说明不要写进 modal；modal 只处理确认、选择或保存闭环。
- 同一 family 内的 CTA 位置必须稳定：详情型页在底部，确认型弹层用双按钮，输入型页面让位给输入区，选择器则以底部提交收尾。

## 7. 实施顺序摘要

- 4j0.22.3：先做 Home 首页纵向主栈，再做 Wallet 固定堆叠与空仓态，最后补资产详情、收款、转账，以及 Home/Wallet 两类 overlay。
- 4j0.22.4：先做全屏选择器，再做 VPN 双卡下单流，最后完成详情型页面的信息表与底部 CTA 结构。
- 4j0.22.5：先做 Discover/Growth/Legal 容器与增长页，再做 Profile 设置主页、安全页和头像选择器 overlay。

## 8. 明确不要做的事

- 不要把首页和钱包页改成多列瀑布流。
- 不要为钱包空仓态单独设计另一种空页面骨架。
- 不要把 VPN 详情页简化成只有列表和按钮的普通表单页。
- 不要把 Discover/Growth/Legal 的容器误做成 Profile 设置列表。
- 不要把高层 picker 或确认流程做成脱离父页面语境的纯黑全屏页。
