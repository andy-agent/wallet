# v2rayNG 加密货币支付集成项目
## 需求审查报告 + 可执行开发实施文档

版本：v1.0  
日期：2026-03-29

---

## 一、执行摘要

基于《项目需求.md》进行审查后，结论如下：

- **方向正确**：将 v2rayNG 与 Marzban 进行支付与订阅联动，业务链路清晰，具备落地价值。
- **方案不完整**：原文更像“高层方案草稿”，尚不足以直接指导 Kimi Code 或工程团队快速开发、联调、验收与部署。
- **存在明显错误和风险**：包括产品命名错误、接口权限设计不合理、链上支付识别方案过于理想化、状态机不完整、数据库设计与 Marzban 现状耦合不当、客户端自动登录与订阅下发逻辑过于粗糙。
- **建议落地路径**：先做 **MVP（SOL + USDT-TRC20 + 单套餐/多套餐 + 新购 + 续费）**，将 ERC20、Webhook、资金归集、节点测试、后台页面等放到第二阶段。

本文件分为两部分：
1. **需求文档审查**：指出正确点、错误点、缺失点、修改建议。
2. **完整开发实施文档**：拆解为 Kimi Code 可严格执行的任务清单、接口契约、数据库、联调、测试、部署、验收步骤。

## 二、原始需求文档审查

## 2.1 做得对的地方

### 2.1.1 项目目标定义正确
原文将目标定义为：
- 套餐展示
- 加密货币支付
- 支付成功后开通代理服务
- 自动下发订阅配置

这是正确的产品主链路，可概括为：
**选套餐 -> 创建订单 -> 链上支付 -> 服务端确认 -> 开通/续费账号 -> 下发订阅 -> 客户端导入节点**。

### 2.1.2 技术拆层思路正确
原文分为：
- Android 客户端（v2rayNG）
- 服务端（Marzban 扩展）
- 区块链支付检测
- 存储 / 缓存 / 队列

这种分层是合理的，后续也应保持该边界，避免把支付逻辑塞进客户端。

### 2.1.3 “私钥不落地客户端”是正确原则
客户端只能拿到：
- 收款地址
- 金额
- 订单号
- 支付状态

绝不能持有平台热钱包私钥，也不能让客户端自行验链并直接决定开通服务。

### 2.1.4 支付与订阅发放解耦思路正确
支付检测成功后，再执行账号开通 / 续费 / 下发订阅，这个因果关系是对的。服务端应是唯一可信源。

## 2.2 明确错误的地方

### 2.2.1 项目名称写错：Mazban -> Marzban
原文多处写成 **Mazban**，正确名称应为 **Marzban**。
- 这不是小问题。
- 错误命名会导致仓库、文档、接口、部署目录、变量名全部混乱。
- 开发阶段必须统一更正为 `Marzban`。

### 2.2.2 /api/plans 使用 admin_token 是错误设计
原文：
`GET /api/plans Authorization: Bearer {admin_token}`

这是不合理的。套餐列表属于客户端公开读取资源，不应要求 admin_token。

正确做法：
- 客户端读套餐应使用公开接口，例如 `GET /client/v1/plans`
- 管理后台维护套餐可使用 `GET /admin/v1/plans`、`POST /admin/v1/plans`
- 必须拆分“客户端接口”和“管理接口”

### 2.2.3 订单创建后再生成地址的流程描述顺序混乱
原图中 “返回订单信息” 和 “生成收款地址” 的时序表达不清晰。

正确流程应为：
1. 客户端提交计划购买信息
2. 服务端创建订单草稿
3. 服务端锁定汇率、生成或分配收款地址 / reference
4. 服务端返回完整支付信息

换句话说：
**客户端拿到的订单响应，必须已经包含最终支付信息**，而不是分两次返回。

### 2.2.4 订单状态设计不清晰
原文状态：
- pending
- paid
- confirmed
- expired
- cancelled

问题：
- `paid` 和 `confirmed` 语义容易重叠
- 缺少 `underpaid`、`overpaid`、`processing`、`failed` 等异常状态
- 缺少“链上已看见但确认数不足”的状态表达

建议状态机：
- `pending_payment`：待支付
- `seen_onchain`：已发现交易，待确认
- `confirming`：确认中
- `paid_success`：已确认且金额匹配
- `expired`：超时
- `cancelled`：人工取消
- `underpaid`：少付
- `overpaid`：超额支付
- `failed`：支付校验失败
- `fulfilled`：账号已开通，订阅已下发

### 2.2.5 “memo 识别订单”不能作为通用强依赖
原文对 SOL 使用 memo 字段识别订单，这是**可选增强**，不能作为唯一识别方式。

原因：
- 并非所有钱包都稳定支持或强制包含 memo
- 钱包 UX 不一致
- 用户可能直接复制地址转账，不带 memo

正确做法：
- **主识别维度**：订单独占收款地址 / reference address
- **辅助识别维度**：memo / reference
- 若走共享地址模型，则必须有 reference 机制且有更复杂对账逻辑

建议 MVP：
- 每笔订单分配独立收款地址，避免共享地址对账复杂度

### 2.2.6 “每笔订单新地址”是对的，但地址池表设计不完整
原文 `payment_addresses` 缺少关键字段：
- chain
- token_contract
- status
- allocated_at
- recycled_at
- last_seen_tx

此外：
- SOL 原生币和 TRC20/ERC20 代币地址模型不同
- 地址池不应只靠 `is_used` 标记
- 需要地址生命周期与对账状态

### 2.2.7 将 auth_token 直接加在 Marzban users 表上风险较高
原文建议直接 `ALTER TABLE users ADD COLUMN auth_token...`

问题：
- 直接改现有用户核心表会增加升级成本
- 后续 Marzban 升级、迁移、兼容性更差
- token 不应作为长期静态字段直接放用户主表

建议：
- 自建 `client_accounts` 或 `client_sessions` 表
- 与 Marzban 用户建立外键关联
- 认证令牌单独管理，便于过期、刷新、吊销

### 2.2.8 返回“user token + subscription_url”并不等于客户端就能无缝登录
客户端需要的是**可拉取订阅的稳定凭证体系**，而不是随意拼一个 JWT。

建议：
- 客户端身份体系与 Marzban 管理端鉴权分离
- 为 App 设计单独的 `client_token`、`refresh_token` 或一次性绑定票据
- 订阅下载可以走：
  - 受保护 API 下载
  - 或签名短链

### 2.2.9 WebSocket 支付状态推送不是 MVP 必需项
对于第一阶段，**轮询足够**。

理由：
- v2rayNG 端集成复杂度更低
- 服务端状态机先稳定最重要
- WebSocket 对移动端保活、重连、页面状态管理要求更高

建议：
- MVP 用轮询（3s / 5s / 8s 退避）
- 第二阶段再考虑 SSE / WebSocket

### 2.2.10 非功能目标过于理想化
原文：
- SOL ≤ 5秒
- TRC20 ≤ 30秒

问题：
这不是系统端单方面能保证的 SLA，受以下影响：
- 公链拥堵
- 节点服务商质量
- 钱包广播速度
- 用户 gas / bandwidth 配置

建议改写为：
- 系统在检测到合格链上交易后，应在 3 秒内完成内部状态更新
- 实际到账确认时间取决于链上网络环境，不承诺固定秒级 SLA

### 2.2.11 “确认数”写法不严谨
原文写：
- SOL: 12 confs
- USDT-TRC20: 19 confs

这属于实现策略，不应直接写死在需求文档。

应改为：
- 按链类型配置 `min_confirmations`
- 可后台调整
- 开发、测试、生产环境分别设置

### 2.2.12 ERC20 作为 P1 是合理的，但不应进入一期核心设计复杂度
MVP 如果先做：
- SOL
- USDT-TRC20

已经足够。
ERC20 会引入：
- gas 费用问题
- 代币精度问题
- 节点与索引器依赖
- 地址管理和监控成本

建议一期文档中只保留“预留能力”，不要把大量接口和数据库复杂度为 ERC20 预留到影响首版交付。

## 2.3 缺失的关键需求

### 2.3.1 缺少“新购”和“续费”的明确区分
目前文档只说“购买套餐”，但业务至少分两种：
1. **新购**：创建新用户并开通订阅
2. **续费**：已有用户延长时长/增加流量

创建订单时必须带上：
- `purchase_type = new | renew`
- renew 时需带现有账号标识

### 2.3.2 缺少“匿名购买 / 免注册购买”策略
要明确：
- 是否允许支付后自动创建账号？
- 账号名 / 邮箱 / 密码如何生成？
- 客户端是否完全免注册？

建议 MVP：
- 采用**免注册购买**
- 服务端自动创建账号
- 返回客户端专用凭证

### 2.3.3 缺少“少付 / 多付 / 误付币种 / 重复支付”处理规则
必须定义：
- 少付：不自动开通，进入异常单
- 多付：可按策略补差或人工处理
- 误付非支持币种：不自动开通
- 同一订单多次支付：第一笔成功后其余进入异常资金池

### 2.3.4 缺少“订单过期后链上到账”的处理策略
订单 15 分钟后过期，但用户 20 分钟后才转账怎么办？

必须定义：
- 默认进入 `late_paid`
- 不自动开通
- 后台人工补单或退款处理

### 2.3.5 缺少“幂等”要求
这些动作必须幂等：
- 支付确认回调 / 轮询任务
- 开通账号
- 续费账号
- 下发订阅
- 客户端重复查询订单状态

否则极易重复开通或重复加时长。

### 2.3.6 缺少“审计日志”要求
后台至少要记录：
- 订单状态变更
- 链上交易识别
- 人工补单
- 用户开通/续费
- 客户端拉取订阅

### 2.3.7 缺少“客户端导入订阅失败”兜底逻辑
支付成功不代表一定能导入成功。
客户端必须支持：
- 重新拉取订阅
- 手动复制订阅链接
- 一键重试

## 三、修订后的目标系统方案

## 3.1 系统边界

### 客户端（v2rayNG 改造）
负责：
- 展示套餐
- 创建订单
- 显示支付二维码 / 地址 / 金额
- 轮询订单状态
- 支付成功后保存凭证
- 拉取并导入订阅

不负责：
- 验链
- 价格换算
- 钱包密钥处理
- 支付成功判定

### 服务端（Payment Bridge + Marzban Integration）
负责：
- 管理套餐
- 创建订单
- 锁定汇率
- 分配收款地址
- 链上监听 / 对账
- 开通 / 续费 Marzban 用户
- 生成客户端凭证
- 输出订阅

### 区块链监听层
负责：
- 查询链上交易
- 识别到账
- 检查金额、币种、确认数
- 回写订单状态

## 3.2 推荐架构

### 推荐不是直接大改 Marzban 核心，而是加一层支付桥接服务
推荐结构：
- `v2rayNG app`
- `payment-bridge service`（新服务，FastAPI）
- `marzban admin api`（已有系统）
- `blockchain watcher workers`
- `postgresql`（推荐）
- `redis`（任务队列 / 幂等锁 / 缓存）

原因：
- 降低对 Marzban 核心侵入
- 便于独立发布支付模块
- 便于后续替换客户端或扩展支付方式

**重要建议**：
如果必须用现有 SQLite，也建议一期只做 PoC。正式上线建议换为 PostgreSQL。

## 四、完整开发实施文档（给 Kimi Code 执行）

## 4.1 开发总原则

Kimi Code 必须严格遵循以下原则：

1. **先后端、后客户端**，先把订单状态机跑通。
2. **先 MVP，再增强**，不要一期实现 ERC20/WebSocket/资金归集。
3. **所有支付相关动作必须幂等**。
4. **所有开通/续费动作必须服务端主导**。
5. **客户端只信任订单状态接口**，不信任本地链上判断。
6. **数据库迁移必须独立可回滚**。
7. **每个任务完成后必须有验证步骤**。

```json
  "client_version": "2.0.15",
  "client_token": "..."
```


- 地址池不足时返回明确错误码

## 阶段 F：链上监听与支付确认

### F-1. 抽象链监听接口
定义统一接口：
- `scan_incoming_transactions(address)`
- `get_transaction_confirmations(tx_hash)`
- `parse_transfer_amount(tx)`
- `match_asset(tx)`

### F-2. Solana 监听实现
**要求**：
- 支持原生 SOL 转账检测
- 可选识别 memo
- 以地址到账为主，memo 为辅

### F-3. Tron 监听实现
**要求**：
- 支持 TRC20 USDT 转账事件查询
- 精度正确处理（通常 6 位）
- 正确识别合约地址

### F-4. Worker 任务
建议任务：
- `scan_pending_orders`
- `confirm_seen_transactions`
- `fulfill_paid_orders`
- `expire_orders`

### F-5. 订单状态机实现

状态转移：
- `pending_payment` -> `seen_onchain`
- `seen_onchain` -> `confirming`
- `confirming` -> `paid_success`
- `paid_success` -> `fulfilled`
- `pending_payment` -> `expired`
- 任意 -> `underpaid` / `overpaid` / `failed`

**硬性要求**：
- 同一 tx_hash 只能绑定一个订单
- fulfillment 必须幂等
- worker 崩溃重启后可恢复扫描

**验证**：
- 用 mock 链服务覆盖足够测试用例
- 用测试网或沙盒做最少 2 条真实链路测试

## 阶段 G：Marzban 集成

### G-1. 抽象 Marzban Client
提供方法：
- `create_user(plan)`
- `renew_user(username, plan)`
- `get_subscription(username)`
- `get_user(username)`

### G-2. 新购流程
当订单进入 `paid_success`：
1. 生成唯一用户名
2. 调用 Marzban 创建用户
3. 设置流量 / 到期时间
4. 获取订阅链接或订阅内容
5. 生成客户端会话令牌
6. 订单置为 `fulfilled`

### G-3. 续费流程
当订单为 renew：
1. 根据 client_token 找到绑定用户
2. 查询 Marzban 用户
3. 按策略续时长/叠加流量
4. 刷新订阅信息
5. 更新订单为 `fulfilled`

**验证**：
- 重复执行 fulfill 不会重复开通
- renew 不会把新购逻辑跑一遍

### H-3. 订阅接口

**返回**：

**验证**：
- token 失效时返回 401
- 客户端可重新刷新订阅

## 八、最终结论

### 8.1 结论
原始需求文档**不是不可用**，但当前状态更适合作为“立项说明”，**不适合作为直接开发文档**。

### 8.2 最大问题
最大问题不是 UI，而是以下四点：
1. 命名错误（Mazban/Marzban）
2. 权限边界错误（admin_token 暴露到客户端链路）
3. 支付识别方案过于理想化（过度依赖 memo）
4. 缺少可执行级别的状态机、异常规则、幂等和任务拆解

### 8.3 可执行建议
建议你把本文件直接作为新的开发基线：
- 第一部分用于纠正文档
- 第二部分用于指导 Kimi Code 分阶段开发
- 一期只做 SOL + USDT-TRC20 + 新购/续费
